Phase 1: High-Impact UI/UX Improvements
Goal: Reduce friction, increase immersion, and make the "AI thinking" time feel shorter.

1. The "Sentence Builder" Interface (Visual Overhaul)
   Instead of a standard text input box and disconnected buttons, create a cohesive "construction" UI.
   Current State: Text Input -> Wait -> Chip Buttons.
   Improvement:
   The Slot Machine Effect: When the user stops typing, do not just pop the chips below. Have the "next word" slot at the end of their sentence animate (fade/slide in).
   Inline Autocomplete: Display the highest probability word as a "ghost text" overlay inside the input field (like GitHub Copilot). Users can swipe right or hit "Tab" to accept the top suggestion immediately without lifting their finger.
   Visual Continuity: The input field and the prediction chips should share the same container styling, making it look like a continuous sentence stream.
2. "Glassmorphism" & Contextual Backgrounds
   Current State: Dark/Light mode theme.
   Improvement:
   Dynamic Gradients: Change the background gradient subtly based on the Tone selected.
   Formal: Cool blues and slate greys.
   Humorous/Casual: Vibrant oranges or purples.
   Sarcastic: Sharp neon accents on black.
   Glass Sheets: Use BlurView (Expo Blur) for the prediction chips so the background flows through them, making the app feel modern and native.
3. Skeletal Loading States & Haptics
   Current State: Likely a spinner or delay while fetching API.
   Improvement:
   Skeleton UI: While waiting for the API (debounce + network time), show pulsating grey "blobs" where the words will appear. This reduces perceived latency.
   Haptic Feedback: Use expo-haptics.
   Light vibration when prediction chips appear.
   Medium click when selecting a word.
   Heavy thud if the user backspaces/deletes.
4. The "Why?" Bottom Sheet
   Current State: "Reason" is part of the chip data but likely cramped in the UI.
   Improvement:
   Long-pressing a prediction chip opens a Bottom Sheet (using @gorhom/bottom-sheet).
   Content:
   Detailed grammatical breakdown (e.g., "Why Subjunctive here?").
   Alternative synonyms.
   Usage examples in other sentences.
   Phase 2: Key Feature Additions (Functionality)
   Goal: Deepen the learning value and retention.
5. Smart Audio (TTS) & Mic Input (STT)
   Why: Language is spoken, not just written.
   Implementation:
   Input: Add a microphone icon. Use expo-speech-recognition (or native plugins) to allow users to speak their input language. The AI predicts the next spoken word.
   Output: Use expo-speech to read the sentence aloud.
   Auto-play: Option to auto-pronounce the word upon selection.
   Speed Control: Allow users to slow down playback (0.5x, 0.75x).
6. "Correction Mode" (The Grammar Police)
   Why: Users will sometimes ignore predictions and type their own (incorrect) words.
   Implementation:
   If the user types a word that wasn't predicted and is grammatically incorrect, the AI should highlight it in red.
   Action: Tapping the red word shows a "Correction Card": "You wrote 'El agua rojo', but 'agua' is feminine in this context. Suggested: 'El agua roja'."
7. Standardized Difficulty (CEFR Levels)
   Why: A slider 1-10 is arbitrary.
   Implementation:
   Replace the slider with CEFR Levels (A1, A2, B1, B2, C1).
   Backend Logic: Inject this into the prompt.
   A1: "Use only top 500 common words."
   C1: "Use idioms, complex compound sentences, and nuance."
8. Spaced Repetition System (SRS)
   Why: The savedPhrases feature is currently a graveyard for data.
   Implementation:
   Implement a lightweight Flashcard system in the Explore tab.
   Algorithm: When a user saves a phrase, schedule it for review (1 day, 3 days, 1 week).
   UI: Tinder-style swipe cards for saved phrases.
   Front: Original Sentence.
   Back: Translation + Pronunciation.
9. Roleplay Scenarios (Contextual Learning)
   Why: "Tone" is good, "Context" is better.
   Implementation:
   Add a "Scenario" dropdown next to Tone.
   Options: Ordering Coffee, Business Meeting, Flirting, Emergency Room, Taxi Ride.
   Prompt Engineering: If "Ordering Coffee" is selected, the specific prediction for "I want..." changes from generic "to eat" -> "a cappuccino" / "a latte."
   Phase 3: Architecture & Backend Improvements
   Goal: Reliability, Speed, and Scalability.
10. Stream the Response (WebSockets or Server-Sent Events)
    Problem: Waiting for the full JSON (Translation + 3 Predictions) creates a delay.
    Solution:
    Although JSON validation is strict, consider streaming the translation first so the user sees immediate feedback, while the predictions (which require more compute) load a split second later.
    Alternative: Use Optimistic UI. If the user types "I want", the app locally predicts "to" (common n-gram) instantly while the AI fetches deeper context.
11. Caching Strategy (Redis or Local)
    Problem: API costs and latency for repeated phrases.
    Solution:
    Hash the input prompt (e.g., "Hello|English|Spanish|Casual").
    Check a local cache (LRU Cache in memory) or a backend Redis instance.
    If the user types "Hello" 100 times, you shouldn't pay DeepSeek 100 times.
